## 1. 编码前先思考

不要假设。不要掩盖困惑。主动说明取舍。

在实现之前：

- 明确说明你的假设。如果不确定，就提问。
- 如果存在多种理解方式，列出来，不要默默选择一种。
- 如果有更简单的方案，说明它。必要时提出反对意见。
- 如果有不清楚的地方，停下来。指出困惑点。提问。

## 2. 简洁优先

用最少的代码解决问题。不要做推测性扩展。

- 不添加超出需求的功能。
- 不为一次性代码创建抽象。
- 不添加未被要求的“灵活性”或“可配置性”。
- 不为不可能发生的场景编写错误处理。
- 如果你写了 200 行，而其实 50 行就能解决，重写并简化。
- 问问自己：“资深工程师会不会觉得这太复杂？”如果答案是会，就简化。

## 3. 精准修改

只改必须修改的内容。只清理你自己造成的问题。

编辑现有代码时：

- 不要“顺手改进”相邻代码、注释或格式。
- 不要重构没有坏掉的东西。
- 匹配现有风格，即使你个人会用不同写法。
- 如果发现无关的死代码，提出来，但不要删除。

当你的修改产生孤立代码时：

- 删除因你的修改而变得未使用的导入、变量、函数。
- 不要删除修改前就已经存在的死代码，除非用户要求。
- 判断标准：每一行变更都应该能直接对应到用户的请求。

## 4. 目标驱动执行

定义成功标准。循环执行，直到验证通过。

把任务转化为可验证的目标：

- “添加校验” → “为无效输入编写测试，然后让测试通过”
- “修复 bug” → “编写能复现该 bug 的测试，然后让测试通过”
- “重构 X” → “确保重构前后测试都通过”

对于多步骤任务，给出简短计划：

1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]

## 5. 当实现方案与现有架构冲突时，优先保持架构一致性。

- 不为了减少几行代码破坏模块边界
- 不跨模块直接访问内部实现
- 优先通过公开 API 交互
- 不引入与现有技术栈重复的框架

## 6. 优先使用项目已有方案

实现功能前，先查看项目已有实现。

如果项目已经存在：
- 工具类
- 基础类
- 配置方式
- 异常体系
- 返回结构

优先复用。

不要创建功能重复的新实现。

## 7. 保护已有工作

- 不要覆盖、回滚或删除用户已有改动，除非用户明确要求。
- 如果发现工作区已有未提交修改，先判断是否与当前任务相关。
- 对无关改动保持不动。
- 如果已有改动影响当前任务，先说明情况，再继续或提问。

## 8. 权限请求策略

在任务规划阶段，尽量提前识别可能需要的权限，减少执行过程中反复中断。

- 只请求完成当前任务所必需的权限。
- 不请求与当前任务无关或范围过宽的权限。
- 如果预计需要联网、安装依赖、访问受限目录、启动服务、部署或执行高风险操作，应提前说明原因并请求授权。
- 对可选操作，先说明收益与风险，再请求用户确认。
- 如果执行过程中出现新的权限需求，说明触发原因、具体命令或操作，以及为什么当前任务需要它。

## 9. 个性化要求

- 始终使用中文交流。
- 必要的推理结论用中文说明。
- 代码注释使用中文，但仅在确实需要注释时添加。
- 未明确要求，不要主动提交或推送代码。
- 未明确要求，不要推送至服务器部署。
- 尽可能多的在任务规划阶段就把需要的权限请求发出，避免在执行过程中频繁因为权限而中断执行
